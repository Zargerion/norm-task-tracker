# Полный контекст требований

## Технологии
- Взяты из codex-admin: NestJS, Prisma, PostgreSQL, Next.js, JWT, Argon2
- Деплой: Docker Compose + Ansible (аналог codex-admin)

## Авторизация
- Только через Telegram-бот
- Telegram Bot Token: в ENV (BOT_TOKEN)
- Супер-админ создаётся сидом
- Остальные — через регистрацию на сайте + подтверждение в боте

## Данные супер-админа (сид)
Задаются только переменными окружения при запуске сида (см. `backend/.env.example`): логин, пароль, числовой Telegram ID; при необходимости имя и фамилия. В репозитории и документации не фиксируются реальные учётные данные.

## Роли
1. **SUPER_ADMIN** — видит всё включая раздел "Пространства", может переключаться между ними
2. **ADMIN** — всё в своём пространстве, включая удаление
3. **MANAGER** — просмотр, создание, обновление
4. **EMPLOYEE** — просмотр, нажатие определённых кнопок (сценарии)

## Пространства (Spaces)
- Скрытый раздел, виден только SUPER_ADMIN
- Каждое пространство изолировано: свои пользователи, проекты, материалы
- Супер-админ добавляет первого ADMIN в пространство
- Можно переключаться между пространствами

## Регистрация пользователей
1. Заполнить форму на сайте: имя*, фамилия*, логин*, пароль*, должность, любимый цвет, описание, телефон, email
2. Система генерирует UUID-код и показывает ссылку на бот
3. В боте ввести UUID-код → бот привязывает TG-аккаунт → одобряет → отправляет magic link
4. После: войти через `/войти` в боте или по логину/паролю

## Telegram-бот
- Токен бота только в `BOT_TOKEN` (ENV), не в коде и не в доках с реальным значением
- Команды: /start, /войти (получить magic link)
- Коды: UUID, одноразовые
- TG ID пользователей хранится в БД

## Разделы системы

### 1. Проекты
- Отображаются в колонке/списке
- Поля: название (короткое), описание (Markdown), иконка (upload файла), цвет (из палитры Genshin), временная шкала (milestones: дата + описание), прикреплённые пользователи
- При наведении — красивое окошко с названием и описанием
- TG-уведомление всем участникам при наступлении milestone

### 2. Пользователи
- Все участники пространства
- Иерархическая пирамида: Admins → Managers → Employees
- Admin может менять роли и описания

### 3. Трекеры (Kanban)
- Селект проекта вверху
- 4 статуса задач: CREATED → ACCEPTED → COMPLETED → APPROVED
- EMPLOYEE: может создавать задачи, двигать до COMPLETED, не может ACCEPT
- MANAGER/ADMIN: могут ACCEPT, могут откатить COMPLETED→ACCEPTED
- После ACCEPTED нельзя откатить в CREATED
- Отслеживание реального времени выполнения (записи в БД)
- Цвет карточки по сложности (часы):
  - 1-3ч: Обычный (серо-белый)
  - 3-5ч: Нормальный (синий)
  - 5-8ч: Уникальный (фиолетовый)
  - 8-20ч: Особый уникальный (яркий фиолетовый)
  - >20ч: Реликвия (тёмно-розовый, блестящий)

#### Карточка задачи
- Название, Описание (Markdown), ориентировочное время, назначенные юзеры
- Прикреплённые файлы (картинки отображать, файлы — ссылка для скачивания)
- "Дополнения" — анонимные Markdown-комментарии (бесконечно)
- При назначении — TG уведомление назначенным юзерам
- При завершении — TG уведомление менеджерам проекта + ссылки Google Meet, Яндекс Телемост
- При откате COMPLETED→ACCEPTED — TG уведомление исполнителям с указанием кто откатил

### 4. Материалы
- Документы в формате Markdown или HTML
- Доступ: публичный (все включая неавторизованных), все участники пространства, точечно по юзерам
- Поля: название, тело (Markdown) или HTML-файл (upload), прикреплённые файлы
- Скачать как HTML
- При Markdown-типе: редактировать текст; при HTML-типе: загрузить файл
