# Роли и права доступа

## Глобальные роли

| Роль | Описание |
|------|---------|
| `SUPER_ADMIN` | Видит всё, переключает пространства, создаёт первых ADMIN |
| (остальные) | Роль определяется в контексте пространства (SpaceRole) |

## Роли в пространстве (SpaceRole)

| Роль | Описание |
|------|---------|
| `ADMIN` | Полный доступ в пространстве, включая удаление |
| `MANAGER` | Просмотр, создание, обновление |
| `EMPLOYEE` | Просмотр, определённые кнопки/сценарии |

## Бизнес-правила для задач

### Переходы статусов

```
CREATED → ACCEPTED  (только ADMIN/MANAGER/SUPER_ADMIN)
ACCEPTED → COMPLETED (все, включая EMPLOYEE)
COMPLETED → ACCEPTED (только ADMIN/MANAGER/SUPER_ADMIN)
ACCEPTED → CREATED   (ЗАПРЕЩЕНО ВСЕМ после принятия)
COMPLETED → APPROVED (только ADMIN/MANAGER/SUPER_ADMIN)
```

### Трекинг времени
- При переходе ACCEPTED→COMPLETED: создаётся `TaskTimeRecord` с `acceptedAt` (берём из текущей записи без completedAt)
- При переходе COMPLETED→ACCEPTED: проставляется `completedAt` в последнюю запись без него → создаётся новая запись с новым `acceptedAt`
- Итоговое время = сумма всех `(completedAt - acceptedAt)` по записям TaskTimeRecord

## Проверка прав в коде

```typescript
// Guards
@SuperAdminOnly()        // только SUPER_ADMIN
@Roles(SpaceRole.ADMIN)  // ADMIN и выше в пространстве
@Roles(SpaceRole.MANAGER) // MANAGER и выше в пространстве
// без декоратора + @UseGuards(JwtAuthGuard) = любой авторизованный участник пространства
@Public()               // без авторизации
```

## Иерархия проверки

1. Является ли пользователь SUPER_ADMIN? → да: всё разрешено
2. Является ли пользователь участником данного пространства? → нет: 403
3. Какая роль пользователя в пространстве? → проверить по матрице
